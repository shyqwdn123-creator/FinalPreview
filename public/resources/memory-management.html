页号 = 逻辑地址 / 页大小

5 碎片特点总结
分页存储碎片特点
✓ 无外部碎片
页框大小固定，任何空闲页框都可分配给任何页，不会产生无法利用的小空闲块
✗ 有内部碎片
进程最后一页往往未填满，剩余空间无法被其他进程使用
内部碎片 ≤ 页大小 - 1
页越大 → 内部碎片可能越大
页越小 → 页表越长 → 开销越大

优点：无外部碎片，内存利用率高

缺点：有内部碎片；页表占用内存空间

折中：页大小的选择需要在碎片和开销之间权衡

分段存储管理

按程序的逻辑结构划分段，每段独立编址，更符合程序员视角

1 核心思想 — 按程序逻辑划分
程序逻辑视图 → 物理内存映射
逻辑视图
代码段 (1024B)
数据段 (2048B)
堆段 (512B)
栈段 (4096B)
→
物理内存
代码段 @0x1000
空闲
数据段 @0x3000
栈段 @0x6000
堆段 @0xA000

程序由若干逻辑段组成，每段长度不等

• 代码段：1024B

• 数据段：2048B

• 堆段：512B

• 栈段：4096B

各段在物理内存中独立放置，不要求连续

2 地址结构 — 段号 + 段内偏移
分段逻辑地址
逻辑地址： <段号, 偏移>
段号： —
段内偏移： —
例：<2, 300> → 段号为2，段内偏移300

分段地址是二维地址：由段号和段内偏移共同确定

段号指定访问哪个段，偏移指定段内的位置

逻辑地址 = <段号, 段内偏移>

与分页不同：分段地址需要用户/编译器显式指定段号

4 分段 vs 分页 对比
对比维度 分页 分段
大小固定（由系统决定）可变（由程序决定）
碎片有内部碎片有外部碎片
地址空间一维（单地址）二维（段号+偏移）
对程序员透明（不可见）可见（需指定段）
共享较难实现容易（按段共享）
保护按页保护按段保护（更自然）
越界检查不需要（页内偏移固定）需要（偏移 < 限界）

分页侧重于系统效率，消除外部碎片

分段侧重于逻辑清晰，方便共享和保护

记忆口诀：页固定无外碎有内碎，段可变有外碎无内碎
5 碎片特点
✓ 无内部碎片
每段按实际需要分配，不会多分配多余空间
✗ 有外部碎片
段长不等，反复分配释放后产生不连续空闲区域

分段与分页的碎片特点正好相反

分段：无内部碎片，有外部碎片
分页：无外部碎片，有内部碎片
Chapter 04

段页式存储管理

先分段再分页，结合两者优点：既保持逻辑清晰，又消除外部碎片

1 为什么要结合
分段的问题
✗ 有外部碎片
段长不等，难以管理
解决方案
在每段内再分页
用分页消除外部碎片
段页式
✓ 保持逻辑清晰
✓ 消除外部碎片
✗ 仍有内部碎片
先分段（保持逻辑结构）→ 每段内再分页（消除外部碎片）

段页式 = 分段 + 分页

先按逻辑分段，再在每段内分页

每个段有自己的页表，段表记录各段页表的起始地址

2 地址结构 — 段号 + 页号 + 页内偏移
段页式逻辑地址
逻辑地址： <段号, 页号, 偏移>
段号： S
页号： P
页内偏移： d
逻辑地址 = 段号S + 页号P + 页内偏移d

段页式地址是三维的

段号确定段 → 页号确定段内页 → 偏移确定页内位置

注意：段页式中，偏移由硬件自动分解为"页号+页内偏移"，对程序员而言地址仍为 <段号, 段内偏移> 形式
3 两级查表 — 段表 → 页表 → 内存

→

Step1: 段号S → 段表 → 页表基址
Step2: 页号P → 页表 → 帧号f
Step3: 物理地址 = f × 页大小 + d

Step 1：取段号，查段表 → 得到该段页表的基址

Step 2：取页号，查内层页表 → 得到帧号

Step 3：帧号 + 偏移 → 物理地址

访存次数：段页式需要 3次访存（查段表 + 查页表 + 访问目标），TLB可加速
